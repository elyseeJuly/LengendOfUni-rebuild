# 全局项目开发文档与归档管理规范 (Global Development & Archive Standard)
> **版本号 (Version)**: V1.0.0  
> **生效日期 (Effective Date)**: 2026-05-20  
> **适用对象 (Target)**: 所有项目开发、AI 协同助手、独立工程师与版本归档  

---

## 📖 一、 概述与核心目的

随着工程和协作体系的不断扩大（多项目并行、AI 助手深度参与），维护一套高度一致、清晰易读且利于检索的文档管理规范是确保项目生命周期顺畅流转的根本。

本规范的核心目的在于：
1. **杜绝文档混乱**：通过科学的命命名前缀和时间排序，使文档在文件管理器或编辑器中实现“自然归类”和“时间线追溯”。
2. **AI 无缝接手**：为接手开发的 AI 助手提供绝对明确的上下文环境、任务交付格式与行为准则。
3. **沉淀历史轨迹**：确保每个重大的审计、规格说明、测试和会话执行都有迹可循，杜绝“代码跑通，思路黑盒”的情况。

## 🛠️ 二、 “分类+日期”双重命名公式

所有项目的开发文档（通常位于 `docs/` 或 `02_Project_Documentation/` 等文件夹内）必须严格遵守以下命名公式：

$$\Large \texttt{[CATEGORY\_PREFIX]\_[YYYYMMDD]\_[DOCUMENT\_NAME].md}$$

### 1. 字段解析说明：
*   **`[CATEGORY_PREFIX]`**：五大全局分类大写前缀之一（参见第三节），用以规范文档属性。
*   **`[YYYYMMDD]`**：文档创建日期或重大修改日期，必须使用 8 位数字。
*   **`[DOCUMENT_NAME]`**：精炼、清晰的英文大写名称，单词之间使用下划线 `_` 隔开。如果是特定国家的补充指令，可在尾部加上 `_CN`（例如 `_DIRECTIVES_V2_CN.md`）。

### 2. 核心优势：
*   **物理天然归组**：按文件名排序时，所有文档会根据所属分类（如 `SPEC_`、`AUDIT_`）自动成组聚集，无需建立繁多的嵌套子文件夹。
*   **时间线完美追溯**：在同类别前缀下，文档会按 `YYYYMMDD` 升序自动从早到晚排列，便于浏览功能演进历史。

## 🏷️ 三、 五大全局分类前缀定义

| 大写前缀 | 英文分类 (Category) | 适用文档范围 | 编写要求与规范 |
| :--- | :--- | :--- | :--- |
| **`SPEC_`** | **Specifications**<br>(规格说明与设计系统) | 核心架构设计、React/TS UI 指令书、美术需求说明书、迁移方案、全局标准文档。 | 必须包含清晰的模块边界、开发规范约束、数据结构接口（Interface）定义及 AI 执行规则。 |
| **`AUDIT_`** | **Audits**<br>(审计报告与系统平衡) | 代码静态审计、叙事与交互心流评估、难度/数值曲线平衡建议、性能优化案。 | 必须列出具体问题的描述、在代码中的具体证据文件/行号，以及明确的解决优化建议。 |
| **`TEST_`** | **Tests**<br>(测试套件与验证用例) | 单元测试用例全覆盖、Headless 自动化试玩指令、自动化测试框架标准、测试补充专案。 | 必须详述测试用例的设计边界、断言逻辑、自动化脚本运行方式，以及测试覆盖率目标。 |
| **`HIST_`** | **Histories**<br>(历史编年史与归档) | 开发日志大编年史、README.md 演进历史、Git 提交流水账、legacy 历史备忘录。 | 主要作为静态归档。要求内容充实，记录关键节点的实施大事件、版本冲突合并日志等。 |
| **`EXEC_`** | **Executions**<br>(活动执行与交付物) | 当前开发会话计划书、待办任务检查清单、阶段性交付验证 Walkthrough 汇报。 | 作为开发执行过程的伴生文档。在会话开始时创建，在任务交付后完成归档记录。 |

## 📝 四、 AI 协同与会话归档管理要求

当 AI 助手深度参与协同开发时，必须遵循以下“三位一体”的执行归档规范。每次重大的功能开发或重构会话，都必须产生并归档以下三份 `EXEC_` 系列文档：

```
[会话启动] ─> 创建 EXEC_[DATE]_PLAN.md (实施方案书)
              │
              └─> 创建 EXEC_[DATE]_TASK.md (任务进度清单)
                  │
[开发验证] ───────> 生成 EXEC_[DATE]_WALKTHROUGH.md (交付验证汇报) ─> [同步并推送至 GitHub]
```

### 1. 实施方案书 (`EXEC_[DATE]_PLAN.md`)
*   **目的**：说明在不修改底层数值/核心公式的原则下，准备如何进行重構或修复。
*   **内容**：说明设计痛点、要修改/新增的文件映射表、AI 执行约束（如“拒绝 window.alert”、“强制使用 Tailwind CSS”等）。

### 2. 任务进度清单 (`EXEC_[DATE]_TASK.md`)
*   **格式**：统一采用 markdown 的 TODO task 列表，如 `[ ]` 未启动、`[/]` 进行中、`[x]` 已完成。
*   **状态同步**：开发过程中，AI 必须随着进度的推进实时更新该文件的 TODO 状态，并在每次转交前将进度情况固化在文件中。

### 3. 交付验证汇报 (`EXEC_[DATE]_WALKTHROUGH.md`)
*   **目的**：向开发者证明代码的健壮性。
*   **硬性指标**：
    *   **测试运行记录**：列出单元测试或集成测试（例如 Vitest/Jest）的运行截图、控制台输出或通过率。
    *   **字节对比审计**：如果是重组/重命名等纯格式调整，必须提供 1:1 字节级的验证，保证原数据完好无损。
    *   **自动化验证**：如果存在 headless 试玩或页面行为分析，必须附带运行日志结果。
    *   **GitHub 同步状态**：详细列出推送目标分支及 remote 状态。

## 🎨 五、 通用文档编写与视觉规范 (Best Practices)

为保持所有文档的高端品质和一致的阅读体验，编写新文档时应严格遵守以下格式要素：

1. **结构化头部信息 (Front Matter)**
   每个文档的顶部必须包含简明扼要的标头，便于读取：
   ```markdown
   # 宇宙群英传 (Legend of Uni) — 游戏优化方案
   > **版本号**: Web 重构版 (03_Web_Rebuild)
   > **基于**: 全系统代码审查 + Headless 自动化试玩实测
   > **生成日期**: 2026-05-17
   ```

2. **活用 GitHub 样式的高级警报 (Alerts)**
   使用通知框高亮关键信息，严禁连续使用或嵌套：
   *   `> [!NOTE]`：用于提供背景上下文或解释。
   *   `> [!TIP]`：用于性能优化建议、最佳实践或提效方案。
   *   `> [!IMPORTANT]`：用于必要要求、AI 关键步骤或强制约束。
   *   `> [!WARNING]`：用于防踩坑、不兼容更改或潜在越界问题。

3. **精准的文件链接与行号锚定**
   引用代码时，严禁使用模糊的描述。必须使用**绝对文件链接**，并精准链接到代码的行范围：
   *   *正确示例*：引用事件管理器的初始化逻辑，请参阅 [GameEventManager.ts](file:///Users/quantumrose/Documents/Emberois/LengendOfUni-rebuild/03_Web_Rebuild/src/core/GameEventManager.ts#L45-L68)。
   *   *优势*：开发者在支持 Markdown 预览的编辑器中可直接“一键点击”直达具体的代码片段，大幅提升审计与修改效率。

4. **主注册表目录同步**
   每个项目应在本地维护一个当前目录下所有文档的主索引文件（如本地的 `SPEC_20260519_DOCUMENTATION_STANDARDS.md`）。
   *   每当 AI 助手或开发者创建了新文档时，**必须第一时间**将新文档的文件链接与简要描述注册到该主文件中。
